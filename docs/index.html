POSITIONING

一种 A2UI 协议。

广义上，ZoneDSL 属于 A2UI（Agent-to-UI）这一类协议——让 agent / 模型的输出，直接变成结构化、可渲染的界面。它要解决的核心问题，和「让 AI 吐出结构化 UI」这件事是一致的：agent 产出 → 解析 → 组件树 → 界面。

A2UI / WHAT

agent
产出即界面

A2UI 的本质是把模型输出映射成 UI。ZoneDSL 里，AI 只需吐出一段带 ::组件 的文本流，前端解析成组件树并渲染成卡片、图表、时间线。

A2UI / HOW

文本优先
而非协议优先

同类方案多走结构化 JSON / 协议下发。ZoneDSL 选了另一条路：文本优先、与 markdown 共生，最小侵入地扩展组件与主题，更贴合 AI 天然的文本输出习惯，也更省 token。

A2UI / EDGE

为流式
而生

面向 AI 流式吐字做了 streamingSafe 保护，属性未闭合前不更新——这正是把 A2UI 落到真实流式对话场景时最容易被忽略、却最关键的一环。

JSON / JSONL 协议 vs ZoneDSL · 五个关键维度
维度 结构化 JSON / JSONL 协议 ZoneDSL(文本优先)
Token 消耗 高——键名/引号/括号占 30-50% 字符 低——只写值,组件名用 :: 前缀标记
Schema 约束 硬——键漏一个 / 类型错一个,整段解析失败 软——未知组件静默丢弃,不闭合的属性跳过,组件始终在场
流式渲染 不友好——JSON 必须闭合才能 parse,中途都是无效状态 行级渐进——一行完整就渲染一行,streamingSafe 兜底半截值
与 markdown 共存 需要额外字段包装 markdown 段(如 {type:"md",content:"..."}) 原生共存——markdown 段照常写,zone 组件用 :: 顶格/围栏穿插
模型心智负担 需记住嵌套 schema、字段名、必填/选填、类型规则 只需记住组件名与几个属性,写法接近自然缩进的伪代码

结构化协议的强项是严格性,适合内部服务间调用;ZoneDSL 的强项是宽容性与流式友好,适合直面用户的 AI 输出场景——两者互补,不是替代。

CAPABILITY

markdown 美化 + 结构化扩展。

对话式 AI 吐出的 markdown 又硬又干，缺卡片、缺图表、缺主题气质。 ZoneDSL 在 markdown 之上叠了一层可扩展的结构化 DSL， 加上一套可定制的主题作用域——同一份内容，切主题即变身。

快速上手

ZoneDSL 的核心使用者是 AI——业务只需要让 agent 会输出、 前端能渲染即可。四步走通: ① 让 AI 学会 skill(必做) → ② 安装到项目 → ③ 修引用路径 → ④ 实现分发器(可选)。

Note · 当前落地平台
当前 Beta 阶段以微信小程序为例讲解接入(towxml 集成、 zone-node 组件、分包目录结构)。ZoneDSL 的 parser 是平台无关的纯 JS (无 DOM 依赖),将来会补上 Web / React Native / Flutter 等端的参考实现, DSL 与组件规范保持一致。

核心链路

ZoneDSL 的工作流水线本质上是四步:

  1. AI 后端流式吐出一段 markdown + zone-DSL 混排文本
  2. 前端 agentChat 页把 content 拿到,交给 buildAst()
  3. towxml/index.js 里的 extractZoneBlocks() 先拆出 zone 块, markdown 走 towxml 原有解析,zone 块交给 zone-dsl/toWxNodes.js 转成组件节点树
  4. 页面渲染层遇到 item.isZone 派发给统一的 <zone-node> 组件, wxml 里 60+ 个 wx:elif 分支各自渲染成对应 zone 组件
TL;DR
你只需要产出 DSL 文本,剩下的解析、组件派发、主题作用域、 流式安全全部由 ZoneDSL 引擎在小程序端搞定。

Hello ZoneDSL

最短的一个 zone 组件:

::text "你好, ZoneDSL"

渲染出来就是一段正文文字。::<组件名> 就是 ZoneDSL 的核心语法。

纯 markdown 用法

ZoneDSL 不排斥 markdown——你完全可以只写 markdown,ZoneDSL 引擎会照常 把它交给 towxml(此处以小程序平台为例) 渲染,得到跟你熟悉的 markdown 一样的结果:

# 一级标题

- 列表项 1
- 列表项 2

| 表头 | 数值 |
| --- | --- |
| A | 1 |

3. 实现 zone-action 分发器

DSL 里 ::button intent=xxx value=yyy 点击会触发 intent,业务方在分发器里实现具体跳转(平台无关,详见 @zonedsl/core/INTENTS.md):

// Web
handleZoneAction(intent, value) {
  switch (intent) {
    case 'search': location.href = `/search?q=${value}`; break
    case 'copy': navigator.clipboard.writeText(value); break
    case 'followup': /* 填输入框 */ break
    // ... 其他 intent 按你项目实现
  }
}
// WeChat: onZoneAction(e) { const {intent,value}=e.detail; ... }
备注
intent 是一种点击意图机制,以下 8 个是常用示例(followup / send-message / search / open-topic / open-tab / open-url / copy / share),宿主可自行扩展平台专属 intent(如电商 open-cart)。详见 @zonedsl/core/INTENTS.md。未识别的 intent 静默降级为纯样式按钮,不会报错。

语法参考

ZoneDSL 一共只有 4 个语法要素:组件声明、属性、子节点、行内高亮。 本页把它们全部列清楚,同时说明和 markdown 的混排方式。

组件声明

顶格两个冒号加组件名,一行一个:

::text "这是一段文字"
::divider
::magazine-cover title="标题"

组件名会在@zonedsl/core/toWxNodes.js 的 COMPONENT_REGISTRY 里查找;查不到默认静默丢弃, 不会渲染出「未支持」占位。

属性 attrs

组件名后面跟属性,格式 key=value,值有引号和裸值两种:

// 引号包裹的字符串
::magazine-cover title="两天刷屏" subtitle="48 小时高光时刻"

// 裸值(单词/数字,无空格)
::grid cols=3 gap=md

// flag 属性(无 = 号,值为 true)
::radio "选项 A" selected
值内引号转义
属性值里如果要出现英文双引号,用 \" 转义,或者用中文引号 「」 「」代替。
换行用 \n(在模板字符串里写 \\n)。

子节点

用 2 空格缩进表示子节点:

::numbered-list "看点在哪"
  item "四姐气场碰撞"
  item "南美世界杯氛围"
  item "新老代际对话"

::data-board cols=4
  item "主榜最高" value="8位"
  item "在榜时长" value="11h+"

子关键字白名单:item / row / field / series / option / step / radio。 不在白名单里的裸词会被当纯文本处理。

main 快捷值

组件名紧跟一个字符串,等价于 main="..."(约定就是主标题/主内容):

// 这两种写法等价:
::card "Q4 核心指标"
::card main="Q4 核心指标"
⚠️ 别重复写
写了 main 就别再写 title:::card "Q4" title="Q4 核心" 是重复。 引号紧跟组件名就是 main,大部分组件会自动把 main 用作 title。

markdown 与 zone 混排

ZoneDSL 支持三种混排:

A · 纯 markdown

如果只需要标题+段落+列表,直接写 markdown。zone-DSL 引擎照常识别。

B · 顶格 :: 混排(推荐 95% 场景)

markdown 正文中穿插顶格 ::,前后各留一行空行:

# 今日热点

三条线索都值得看。

::magazine-cover tag="HOT SPOTS" title="今日三焦点"

## 焦点一: 盛夏夜

正文继续...

C · 围栏 ```zone 块

用 ```zone ... ``` 显式围出一段 zone,parser 只处理围栏内的部分:

```zone
::magazine-cover
  tag="HOT RECAP" title="**48小时刷屏**"

::chapter "01" title="先看数据"
::data-board cols=2
  item "主榜最高" value="8位"
  item "在榜时长" value="11h+"
```

markdown 段落继续,不会被 zone 影响。
两种边界的等价性

B · 顶格 :: 混排 和 C · ```zone 围栏 在渲染结果上完全等价 —— parser 会把两种边界都替换成同一个 zone-dsl-placeholder 占位,再统一走 zone-DSL 引擎。

选哪种取决于场景:AI 流式吐字用 B(无围栏,吐一行渲一行体验最好); 正式文档 / Web / 长内容 用 C(边界最清晰,不怕缩进错位)。

围栏 meta 传主题(可选)

用围栏时可以直接在开栏行写 theme=xxx,parser 会把它当作块首 ::theme 处理:

```zone theme=literary
::magazine-cover title="梅雨季"
::chapter ...
```

等价于:

```zone
::theme literary
::magazine-cover title="梅雨季"
::chapter ...
```

行内高亮(title/subtitle 通用)

ZoneDSL 在标题/副标题里支持三档行内高亮,和 markdown 语法一致:

语法视觉用途
**text** 亮金黄块 + 深炭字 主强调,视觉重量拉满
~~text~~ 橙色字 + 底部半高浅金橙块 次强调,马克笔效果
==text== title:深炭字 + 橙下划线
subtitle:橙色加粗字		 轻强调

主题声明

三种入口都可以,一个 zone 块只需要写一次:

// 入口 A:magazine-cover 属性
::magazine-cover theme=literary title="..."

// 入口 B:顶格伪组件(推荐用于 markdown 接管系主题)
::theme purple

// 入口 C:围栏 meta
```zone theme=data
::chapter ...
```

三种声明等价,优先级 A > C > B。不写就是默认 editorial。

意图与交互

ZoneDSL 有一套意图白名单让 AI 输出的按钮 / 表单能真的触发业务动作,而不是只做视觉。核心是 ::button intent=xxx value=yyy 和其它交互组件 (checkbox / radio / quiz / select),用户点击时事件逐级冒泡到宿主的 onZoneAction 分发器。

事件冒泡链

用户点击一个组件后,事件按下面的路径逐级传上去,最终到达业务方的分发器:

// 组件层                → 消息层                → 页面层
zone-node.onButtonAction   →  message-item.onZoneAction  →  agentChat/index.js.onZoneAction(e)

// 到达页面层时 e.detail 结构固定为:
{
  intent: 'followup',      // 白名单里的 intent
  value:  '具体参数',       // intent 的 payload
  source: 'button',          // 触发组件类型
  ext:    { ... }              // 可选扩展字段(如 checkbox-group 的 items 数组)
}

业务侧分发器模板

onZoneAction(e) {
  const { intent, value, source } = e.detail
  switch (intent) {
    case 'followup':
      // 把 value 塞进输入框,自动发一条追问
      this.sendFollowup(value)
      break
    case 'search':
      wx.navigateTo({ url: `/pages/search/index?q=${encodeURIComponent(value)}` })
      break
    case 'copy':
      wx.setClipboardData({ data: value })
      break
    default:
      // 未实现的 intent 静默忽略(不会报错,组件降级为纯样式)
      console.log('[zone] unhandled intent', intent, value)
  }
}

其它交互组件

除了 ::button intent=...,还有一批完全独立于 intent 的交互组件——它们的 state 由 zone-node 内部维护,不需要业务侧处理:

  • ::tabs / ::accordion —— 展开/收起,组件内部 state
  • ::checkbox / ::checkbox-group —— 多选,change 事件可冒泡
  • ::radio / ::radio-group —— 单选,同组互斥
  • ::select / ::textarea —— 表单输入
  • ::quiz —— 单选题,点击选项高亮

需要把 checkbox/radio 的结果发给业务,再套一个 ::button intent=followup value="...",由用户点确认按钮触发。

常见错误

  1. 顶格 :: 前后不留空行 —— towxml 用空行作为 zone 块的边界,紧贴段落会解析错乱
  2. 子项用 Tab 或 4 空格缩进 —— parser 按缩进层级识别子节点,必须用 2 空格
  3. 属性值里嵌英文双引号 —— 用 \" 转义或改中文引号
  4. attrs 里直接按回车 —— parser 按行 tokenize,attrs 内真实换行会截断组件,用 \\n 表示换行
  5. 写不存在的组件名 —— 默认静默丢弃,不会报错,但你的组件不会出现。请参考 组件全览
  6. 网格深嵌套 3 层以上 —— row/col/grid 深嵌小程序会卡顿,建议 2 层内

主题声明语法

三种入口等价,一个 zone 块只需要声明一次。优先级 A > C > B。不写就走默认 editorial。

入口 B · 顶格 ::theme(无封面时)

::theme 是伪组件,只影响主题,不会渲染任何 UI:

::theme purple # 备忘便签 正文段落,可以用 **加粗** 强调重点。

zone-node 组件是怎么工作的

zone-node 是 zone-DSL 的统一派发组件 —— 不管你写的是 ::text 还是 ::magazine-cover, 最终都会被 towxml 解析成同一个 <zone-node> 节点, 再由内部的 60+ 个 wx:elif 分支按 tag 派发到对应模板。

三个关键设计:virtualHost 让 host 透明、styleIsolation 跨作用域命中、addGlobalClass 让主题类落到子组件。三合一让 zone-node 变成一个视觉上完全消失的容器。

数据流

从 AI 吐字到最终渲染,整个链路只有三步:

Parse
2ms
DSL → AST
Transform
1ms
AST → nodes
Render
8ms
nodes → DOM

为什么不冲突

  • markdown 的 # 标题、**加粗**、- 列表 照常走
  • zone 语法用 :: 前缀,和 markdown 的语法字符完全分开
  • parser 先扫 zone 块占位,再让 towxml 处理剩下的纯 markdown
FENCE · 边界语法

Zone 围栏 · 两种边界

Zone 块跟 markdown 段落穿插时,parser 需要知道「哪里是 zone、哪里是 markdown」。 提供两种等价语法:```zone 围栏(web 惯例,显式) 和 :: 顶格 连续行(小程序 AI 常用,隐式)。上下两段渲染完全一致 —— 用哪种取决于场景。

EXAMPLE 01 · MAGAZINE COVER

杂志封面 · 复盘头图

一行 DSL 生成一张完整杂志式封面卡:tag / title(含高亮) / subtitle / stats。 ::magazine-cover 是 preset 层的招牌组件。

DSL
::magazine-cover
  tag="HOT RECAP · 2026.07"
  title="林晚\n**盛夏夜名场面**"
  subtitle="48 小时刷屏,4 大场面轮番出圈"
  stats="19 条热议,5 条冲主榜,11h+ 最长在榜"
  bg=light

::theme purple
